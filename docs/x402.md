# x402 payments

x402 turns the dormant HTTP `402 Payment Required` status code into a working
machine-to-machine settlement layer. A server replies `402` with payment terms;
the client settles and retries with an `X-PAYMENT` envelope. ProwlFi adds the
privacy layer: the payment lands at a one-time stealth address, not a fixed,
public account.

## The flow

1. An agent calls a vendor endpoint.
2. The vendor responds `402 Payment Required` with terms — amount, token,
   recipient meta-address, and x402 version.
3. ProwlFi derives a fresh stealth address from the meta-address and signs an SPL
   transfer authorization.
4. The agent retries with the signed envelope in the `X-PAYMENT` header.
5. A facilitator confirms settlement; the vendor serves the resource and the
   stealth address vanishes into the graph.

The on-chain footprint is one transfer to a one-time address plus one
announcement. Nothing links the buyer to a vendor-facing public account; the
receipt remains verifiable to the buyer and recoverable to the operator via the
viewing key.

## Surfaces

The same flow is available three ways:

- `agent.payX402(req)` in [`@prowlfi/sdk`](../packages/sdk)
- `prowl.x402_pay` in [`@prowlfi/mcp-server`](../packages/mcp-server)
- `POST /v1/x402/pay` on the REST API
